# Task 3.5: Implement Hub Client Fallback

## Status: COMPLETED

## Objective
Implement Phase 3 Hub Client Fallback in `pkg/hubclient/`. The client should now prefer `/api/v1/projects` paths but automatically fall back to `/api/v1/groves` on 404 errors to maintain compatibility with older Hub versions.

## Changes

### 1. Hub Client Fallback Mechanism
- Modified `pkg/hubclient/client.go` to add common request helper methods: `get`, `getWithQuery`, `post`, `put`, `patch`, and `delete`.
- Implemented 404 fallback logic in these helpers: if a request to a path containing `/projects` returns a 404, it retries once by replacing `/projects` with `/groves`.

### 2. Service Implementation Updates
- Updated all service implementations in `pkg/hubclient/` to use the new helper methods instead of calling the transport directly.
- Updated path construction in `AgentService`, `ProjectService`, `GCPServiceAccountService`, `ScheduledEventService`, and `ScheduleService` to use `/api/v1/projects` as the primary prefix.
- Updated `RuntimeBrokerService` to use `/api/v1/runtime-brokers/*/projects`.

### 3. Data Compatibility
- Updated several request and response types to handle dual `projectId` and `groveId` support via `MarshalJSON` and `UnmarshalJSON` (extending the work from Task 3.2):
    - `CreateAgentRequest`
    - `ScheduledEvent`
    - `Schedule`
    - `AgentMessage`
    - `Notification`
    - `Subscription`
    - `CreateSubscriptionRequest`

### 4. Verification
- Added `TestFallback` in `pkg/hubclient/client_test.go` to explicitly verify the 404 retry mechanism.
- Updated existing tests in `client_test.go` to expect `/api/v1/projects` paths.
- Fixed a regression in `pkg/hubclient/types_test.go` where a test case was incorrectly providing both `id` and `groveId` and expecting the latter to win (violating precedence rules).
- All tests in `pkg/hubclient/` pass.
