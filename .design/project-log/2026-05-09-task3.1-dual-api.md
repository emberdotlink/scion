# Project Log - 2026-05-09 - Phase 3 REST API Dual-Path Registration

## Task Summary
Implemented Phase 3 of the Grove to Project rename, which involves supporting dual-path registration for the REST API. Both `/api/v1/groves/...` and `/api/v1/projects/...` paths are now supported and route to the same handlers.

## Changes

### Hub Server
- Modified `pkg/hub/server.go` to register `/api/v1/projects`, `/api/v1/projects/register`, and `/api/v1/projects/` routes as aliases to their `groves` counterparts.
- Updated `pkg/hub/handlers.go`:
    - `handleProjectRoutes`: Updated to handle both `/api/v1/projects/` and `/api/v1/groves/` prefixes when extracting the project ID and subpath.
    - `handleProjectByID`: Updated to handle both prefixes (deprecated endpoint but updated for completeness).
    - Replaced `NotFound(w, "Grove")` with `NotFound(w, "Project")` across multiple handlers for consistency with the new naming.
- Updated `pkg/hub/project_webdav.go`:
    - Cleaned up dead code in `handleProjectWebDAV` and ensured it correctly reconstructs the prefix for WebDAV operations regardless of whether `/groves/` or `/projects/` is used.
- Updated `pkg/hub/web.go`:
    - Updated `resolveAPIPath` to support `/projects` and `/projects/{id}` paths for SSR hydration.

### Utilities
- Updated `pkg/util/logging/request_log.go`:
    - Added `/api/v1/projects/` to `HubPathPatterns` and `BrokerPathPatterns` to ensure request logging correctly extracts project IDs from the new paths.

## Verification
- Ran `make build` to ensure the project still compiles.
- Internal routing logic was updated to be prefix-agnostic where possible.
- Existing tests (which use `/api/v1/groves`) should continue to pass as those routes were preserved.

## Observations
- The codebase already used "Project" terminology internally in many places (e.g., `handleProjects`, `store.GetProject`), which made the aliasing straightforward.
- Some hardcoded path parsing was updated to be aware of the two possible prefixes.
