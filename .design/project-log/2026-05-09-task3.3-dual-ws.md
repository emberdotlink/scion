# Project Log: Phase 3 WebSocket Protocol Dual-Field Support

**Date:** 2026-05-09
**Task:** 3.3 Dual WS Protocol Fields

## Overview
Implemented Phase 3 of the WebSocket Protocol Dual-Field Support. This task involved updating the WebSocket protocol definitions to support both legacy 'Grove' fields and new 'Project' fields to ensure backward compatibility during the rename transition.

## Changes

### `pkg/wsprotocol/protocol.go`
- Updated `ConnectMessage` to include both `Groves []string` (legacy) and `Projects []string` (new).
- Updated `StreamOpenMessage` to include both `GroveID string` (legacy) and `ProjectID string` (new).
- Implemented custom `MarshalJSON` and `UnmarshalJSON` for both structs to handle fallback logic:
    - On Unmarshal: If the new field is empty but the legacy field is present, populate the new field from the legacy one, and vice versa.
    - On Marshal: Ensure both fields are populated if either is present.
- Updated factory functions `NewConnectMessage` and `NewStreamOpenMessage` to populate both fields.

### `pkg/wsprotocol/protocol_test.go`
- Added `TestDualFieldSupport` to verify that the fallback logic works correctly for both `ConnectMessage` and `StreamOpenMessage` during JSON unmarshaling.

### `pkg/hub/controlchannel.go`
- Updated `handleMessage` to unmarshal `ConnectMessage` and log the projects served by the broker (when debug logging is enabled).
- Renamed `groveID` parameter to `projectID` in `OpenStream` methods for consistency with the new naming convention.

### `pkg/hub/pty_handlers.go`
- Renamed `groveID` field to `projectID` in `PTYSession` struct.
- Updated `newPTYSession` and `Run` method to use the new `projectID` naming.

### `pkg/runtimebroker/controlchannel.go`
- Renamed `groveID` field to `projectID` in `StreamHandler` struct.
- Updated `handleStreamOpen` and `handlePTYStream` to use the new `projectID` naming.

## Verification

- Ran `go test ./pkg/hub/...` and `go test ./pkg/runtimebroker/...` to ensure no regressions in control channel or PTY functionality.
